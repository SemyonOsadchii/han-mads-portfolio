# Reflection

Revision note: the original odd values such as `420`, `42`, and `67` were useful as exploratory probes, but they are not a good use of time for the final long training pass. The notebook now keeps that exploratory setup as reference and adds a focused long search with cleaner powers-of-two choices and a fixed `30`-epoch budget.

For this chapter I wanted to understand whether a simple dense network on Fashion-MNIST would benefit more from extra width, extra depth, or longer training. My first hypothesis was that increasing the number of hidden units would help at the start, but that after a certain point the gains would become small. I also expected that training for more epochs would be more useful than just stacking another hidden layer, because this is still a relatively simple model on image data that has already been flattened.

To test that idea, I started with a structured grid search over `units1` and `units2`. I kept the optimizer, learning rate, depth, train steps, and validation steps fixed so that the comparison stayed fair. I used mostly clean values such as `64`, `128`, and `256`, because powers of two make patterns easier to see in a grid. At the same time, I also added `420` because I wanted one deliberately odd value to check whether the trend only looked good because the grid was too neat.

The best grid result came from `units1=420` and `units2=256` with Adam, which reached about `81.15%` validation accuracy after three epochs. That already suggested that wider layers were helping. After that, I ran follow-up experiments around the best region. I tested more epochs, a deeper model, AdamW, RMSprop, and different batch sizes including `42` and `67`.

The strongest result came from the same `420/256` architecture with Adam and `7` epochs. That reached about `82.24%` final validation accuracy, with the best intermediate validation accuracy going even higher during training. This supports the idea that once a good configuration is found, training a little longer can be more effective than changing the architecture again. In contrast, the deeper model reached a strong intermediate score but finished lower, which suggests that the extra layer did not give a stable improvement in this setup.

I also found it useful to compare the more unusual settings. The batch sizes `42` and `67` did not completely break performance, which shows that the model is not extremely fragile, but neither of them beat the strongest baseline. RMSprop with a learning rate of `6.7e-4` came surprisingly close, while AdamW with `4.2e-4` was clearly weaker here. That taught me that "trying something funky" can be useful, but it should still be connected to a clear question. Otherwise it becomes random experimentation instead of a proper scientific process.

One lesson from this chapter is that factors of two are useful because they give a clean and efficient scan of the search space. They make it easier to spot large-scale trends, especially in a heatmap. The downside is that they can hide interesting behavior between those points. Adding one unusual value such as `420` gave me a way to test that limitation without making the notebook chaotic.

If I would continue this experiment, I would keep the best width setting and then study learning rate and epoch scheduling in more detail. I would not immediately expand to a huge search space, because this chapter already showed that a small, well-chosen set of experiments can answer useful questions. For me, the main takeaway is that hyperparameter tuning works best when I combine structure with curiosity: start with a clean grid, make a hypothesis, and then use targeted follow-up runs to explain what I see.
